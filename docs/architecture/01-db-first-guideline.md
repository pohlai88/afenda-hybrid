## 1 Purpose

This document is the **authoritative reference** for how AFENDA models, stores, and evolves data in PostgreSQL. It defines what _DB-first_ means in practice, establishes the boundary between core and domain schemas, and provides concrete, enforceable rules for tenancy, integrity, migrations, testing, operations, security, and observability.

This guideline uses **Drizzle ORM** (v1.0+) as the TypeScript schema language and the **built-in `drizzle-orm/zod`** integration for runtime validation schemas.

Every engineer who touches a migration file, writes a query, or designs a new domain table **must** follow this guideline. Deviations require an Architecture Decision Record (ADR) approved by the relevant Schema Owner and Migration Approver.

## 2 Principles

| #   | Principle                          | What it means in practice                                                                                                                                                                                                  |
| --- | ---------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| P1  | **Database as Source of Truth**    | The primary data model, constraints, and history live in PostgreSQL. Application layers are consumers, not owners, of structural invariants.                                                                               |
| P2  | **Minimal Core, Stable Contracts** | The `core`, `security`, and `audit` schemas remain small and change infrequently. Domain schemas own their own models and evolve independently.                                                                            |
| P3  | **Enforce Invariants in the DB**   | Prefer foreign keys, `CHECK` constraints, `UNIQUE` constraints, and transactions over application-only validation. The database is the last line of defence.                                                               |
| P4  | **Versioned Migrations Only**      | Every structural change is a versioned migration script committed to Git. No ad-hoc DDL in production.                                                                                                                     |
| P5  | **Operational First**              | Every schema decision must account for operational cost, rollback plan, and monitoring requirements before approval.                                                                                                       |
| P6  | **Pragmatic Normalization**        | Use third normal form (3NF) for master data. Allow denormalized read models or materialized views where reporting performance demands it.                                                                                  |
| P7  | **TypeScript as Schema Language**  | Drizzle ORM table definitions are the canonical schema source; raw SQL is generated by `drizzle-kit generate`, not hand-written. Schemas are committed as `.ts` files; migrations are committed as generated `.sql` files. |

## 3 Schema Organization

### 3.1 Three-Tier Model

AFENDA organizes schemas into three tiers:

| Tier       | Name                      | Purpose                                                                                                                    |
| ---------- | ------------------------- | -------------------------------------------------------------------------------------------------------------------------- |
| **Tier 1** | **Foundational (core)**   | Tenants, organizations, regions, locations, settings. Rarely changes. Everything else references this.                     |
| **Tier 2** | **Infrastructure Shared** | `security`, `audit`, `observability` — cross-cutting concerns consumed by all domain schemas. References core.             |
| **Tier 3** | **Domain / Module**       | `hr`, `finance`, `projects`, … — each split into **fundamentals/** (master data) and **operations/** (transactional CRUD). |

```
PostgreSQL cluster
├── core        — Tenants, organizations, regions, locations, tenant settings (Tier 1)
├── security    — Users, roles, service principals, authentication linkage (Tier 2)
├── audit       — Append-only audit trail and retention metadata (Tier 2)
├── observability — Traces, metrics metadata (Tier 2)
│
├── hr          ─┐
├── finance      │  Domain schemas (Tier 3) — fundamentals + operations
├── projects     │
└── ...        ──┘
```

### 3.2 Domain Schema Internals

Every domain schema is logically split into two layers:

| Layer                   | Purpose                                                                                       | Example tables                          |
| ----------------------- | --------------------------------------------------------------------------------------------- | --------------------------------------- |
| **Domain Fundamentals** | Master data and structural invariants                                                         | `hr.employees`, `hr.departments`        |
| **Domain Operations**   | Transactional CRUD that may require partitioning, retention policies, or specialized indexing | `hr.attendance_logs`, `hr.payroll_runs` |

Schema definitions live in TypeScript; Drizzle Kit generates SQL migrations from them.

  src/

    schema-platform/     — core/, security/, audit/, observability/; Drizzle Kit entry `index.ts` re-exports HCM from schema-hrm
      core/              — tenants.ts, organizations.ts, regions.ts, locations.ts, _relations.ts

      audit/             — auditTrail.ts, retentionPolicy.ts, CUSTOM_SQL registry
      observability/     — traces.ts

### 3.4 Naming and Ownership Rules

**Schemas**

- Lowercase, short, stable names (`core`, `hr`, `finance`).
- Never rename a schema after it reaches production without an ADR.

**Tables**

- Fully qualified: `{schema}.{singular_or_plural_noun}` (choose one convention per schema and stick with it).
- Column names in TypeScript use **camelCase**. The `casing` option on `drizzle()` must match **physical** column names: `snake_case` maps TS fields to `snake_case` SQL identifiers; `camelCase` matches quoted mixed-case columns produced by some migration histories (see **db.ts** above).

**Primary Keys**

- Use surrogate `{entity}Id` in TypeScript. Prefer `.generatedAlwaysAsIdentity()` on `integer()` (see Section 4.2). The SQL column name follows your Drizzle/migration casing convention (`tenantId` vs `tenant_id`).

**Ownership**

Every schema must have two named roles documented in the schema registry:

| Role                   | Responsibility                                               |
| ---------------------- | ------------------------------------------------------------ |
| **Schema Owner**       | Approves design changes; accountable for data model quality. |
| **Migration Approver** | Reviews and signs off every migration before it merges.      |

### 3.5 Module Boundary Rule

Cross-schema foreign keys introduce coupling. The rules are:

1. **Within the same schema** — foreign keys are encouraged (inline `.references()` or standalone `foreignKey()`).
2. **From a domain schema to `core` or `security`** — always allowed (these are stable contracts). Use standalone `foreignKey()` for cross-schema references (AFENDA convention).
3. **Between two domain schemas** — allowed **only** with:
   - Explicit written approval from both Schema Owners.
   - A documented rationale in the migration's header comment.
   - An entry in `docs/architecture/cross-domain-fks.md`.

### 3.6 Domain module package pattern (Zod, `CHECK`, and documentation)

Beyond **fundamentals/** and **operations/** (§3.2), mature domain packages should close the loop between **PostgreSQL**, **Drizzle**, **Zod**, **`defineRelations`**, **documentation**, and **tests**. The **`learning`** schema is the reference implementation (`packages/db/src/schema-hrm/learning/`). **`payroll`** and **`benefits`** also ship `_zodShared.ts`; other domains should converge on the same pattern.

#### Building blocks

| Artifact                                            | Purpose                                                                                                                                                                                                                            |
| --------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `_schema.ts`                                        | Single `pgSchema("…")`; docblock lists cross-schema dependencies and domain boundaries (including what this schema is _not_).                                                                                                      |
| `_relations.ts`                                     | One `defineRelations(…)` graph for RQB; use **`alias`** when two foreign keys target the same table (e.g. approver vs subject `employeeId`).                                                                                       |
| `_zodShared.ts`                                     | **Single source of truth** for domain numeric bounds (reuse in Zod and in Drizzle `check()` via shared constants), domain-specific **cross-field refinements** chained with `.superRefine`, and re-exports of shared wire helpers. |
| `zodWire.ts` (`packages/db/src/_shared/zodWire.ts`) | **Cross-domain** Postgres `date` / `timestamptz` wire schemas, `nullableOptional`, `isoDateWireString`, `parseUnknownToEpochMs`, etc. Import here first; domain `_zodShared` files re-export for local ergonomics.                 |
| Domain `README.md`                                  | Layout table, cross-schema dependency matrix, numbered conventions, **cross-field refinement catalog** (each rule: Zod-only vs DB-backed), and `pnpm vitest run …` lines for `*-zod.test.ts` files.                                |

#### Conventions

1. **Bounds** — Define a `{domain}Bounds` object (or equivalent) and reference it in both `createInsertSchema` / `createUpdateSchema` overrides and `check("…", sql\`…\`)` so limits cannot drift between layers.
2. **Enums** — Keep one `const` tuple; `schema.enum("…", [...tuple])` for Postgres; for insert/update DTOs prefer **`z.enum(tuple)`** aligned with that tuple (see §5.2). Do not rely on `createSelectSchema(pgEnum)` alone for write models if it obscures the API’s allowed string union.
3. **Cross-field rules** — Implement shared refinements once; document whether each rule is mirrored by **`CHECK`** in migrations. Any literal in SQL that must match TypeScript (e.g. a threshold percentage) requires a **named exported constant** and a comment pointing to the migration constraint name.
4. **Partial updates** — Document on each `*UpdateSchema` whether nullable columns support explicit SQL `NULL` via `.nullable().optional()` (patch clears) vs plain `.optional()` (omit-only).
5. **Indexes** — Tenant-leading indexes; partial **unique** indexes with `deletedAt IS NULL` where soft-deleted rows exist; optional **filtered** indexes for recurring reporting queries when justified by access patterns (see `learning` operations tables).
6. **Table docblocks** — Note circular FK handling (custom SQL migrations), status/lifecycle semantics, and service-layer duties (e.g. `createdBy` / `updatedBy` on `auditColumns`).

## 4 PostgreSQL Primitives Reference

This section maps every core PostgreSQL structural primitive to its Drizzle ORM API. Use it as the single source of truth for expressing DB concepts in TypeScript.

### 4.1 Named Schemas (`pgSchema`)

Every tier gets a `pgSchema()` declaration. Tables, enums, and sequences are then scoped to that schema.

```typescript
import { pgSchema } from "drizzle-orm/pg-core";

export const coreSchema = pgSchema("core");
export const securitySchema = pgSchema("security");
export const hrSchema = pgSchema("hr");
```

- **Schema-scoped tables**: `coreSchema.table("tenants", { ... })`
- **Schema-scoped enums**: `coreSchema.enum("tenant_status", ["ACTIVE", "SUSPENDED", "CLOSED"])`
- **Schema-scoped sequences**: `coreSchema.sequence("invoice_seq", { ... })`

**Two equivalent forms:**

```typescript
// Form 1: Import-based (most common)
import { integer, text } from "drizzle-orm/pg-core";

export const tenants = coreSchema.table("tenants", {
  tenantId: integer().primaryKey().generatedAlwaysAsIdentity(),
  tenantCode: text().notNull().unique(),
});

// Form 2: Callback-based (alternative)
export const tenants = coreSchema.table("tenants", (t) => ({
  tenantId: t.integer().primaryKey().generatedAlwaysAsIdentity(),
  tenantCode: t.text().notNull().unique(),
}));
```

Both forms are equally valid; choose one and use it consistently within a schema.

### 4.2 Primary Keys

- **Single (recommended)**: `.primaryKey().generatedAlwaysAsIdentity()` on `integer()` — uses PostgreSQL `GENERATED ALWAYS AS IDENTITY` (SQL standard, preferred over serial/bigserial).
- **Single (legacy)**: `.primaryKey()` on `bigserial()` / `serial()` — still valid.
- **Composite**: `primaryKey({ columns: [t.col1, t.col2], name: "pk_name" })` in the table's third argument.

**AFENDA convention**: Surrogate `{entity}Id` (camelCase in TS, auto-mapped to `{entity}_id` in DB via `casing: 'snake_case'`) using `.generatedAlwaysAsIdentity()`.

### 4.3 Foreign Keys

- **Inline**: `.references(() => parentTable.column)` for simple single-column FKs within the same schema.
- **Inline with actions**: `.references(() => t.col, { onDelete: 'cascade', onUpdate: 'cascade' })`
- **Standalone**: `foreignKey({ columns: [...], foreignColumns: [...], name: "fk_name" })` for composite or self-referencing FKs.
- **Self-referencing**: use `AnyPgColumn` — `.references((): AnyPgColumn => table.id)` or standalone `foreignKey()`.
- **Actions**: `onDelete` / `onUpdate`: `'cascade' | 'restrict' | 'no action' | 'set null' | 'set default'`.

**AFENDA convention**: Use standalone `foreignKey()` for **cross-schema** references; inline `.references()` is fine within the same schema.

### 4.4 Indexes

- **Regular**: `index("idx_name").on(t.column)` — btree by default.
- **Unique**: `uniqueIndex("uq_name").on(t.column)`
- **Composite**: `index("idx_name").on(t.col1.asc(), t.col2.nullsFirst())`
- **Partial (WHERE)**: `index("idx_name").on(t.col).where(sql\`condition\`)`
- **Using (non-btree)**: `index("idx_name").using("gin", t.col.op("jsonb_ops"))` or `"gist"`, `"hnsw"` for pg_vector.
- **Concurrently**: `index("idx_name").on(t.col).concurrently()` — for zero-downtime; must be in a separate non-transactional migration.
- **With options**: `.with({ fillfactor: '70' })`

**AFENDA convention**: Index `tenantId` as the leading column on tenant-scoped tables; name indexes `idx_{table}_{columns}`.

### 4.5 Constraints

- **NOT NULL**: `.notNull()` — AFENDA default: every column is `.notNull()` unless explicitly justified.
- **UNIQUE**: `.unique()` inline, or composite `unique().on(t.col1, t.col2)`; use `.nullsNotDistinct()` for PG 15+.
- **CHECK**: `check("chk_name", sql\`${t.status} IN ('ACTIVE','SUSPENDED','CLOSED')\`)` in the table's third argument.
- **DEFAULT**: `.default(value)`, `.defaultNow()`, `.defaultRandom()`, or `.default(sql\`expression\`)`
- **GENERATED**: `.generatedAlwaysAs(sql\`expression\`)` for computed columns (stored/virtual per Section 4.11).

### 4.6 Enums (`pgEnum`)

- **Public schema**: `export const statusEnum = pgEnum("tenant_status", ["ACTIVE", "SUSPENDED", "CLOSED"]);`
- **Named schema**: `coreSchema.enum("tenant_status", ["ACTIVE", "SUSPENDED", "CLOSED"]);`
- **Column usage**: `tenantStatus: statusEnum().notNull().default("ACTIVE")`

Prefer enums over CHECK-based string validation when the value set is stable and shared across tables. Use CHECK for table-local validation.

### 4.7 Relations (Drizzle soft relations)

- **One-to-one**: `relations(table, ({ one }) => ({ profile: one(profiles, { fields: [table.profileId], references: [profiles.id] }) }))`
- **One-to-many**: `relations(table, ({ many }) => ({ posts: many(posts) }))`
- **Many-to-many**: Junction table with `primaryKey({ columns: [t.userId, t.groupId] })`
- **Disambiguation**: use `relationName: "author"` when multiple relations exist between the same two tables.
- **Cross-schema**: relations work across `pgSchema` boundaries.

**Important**: `relations()` is application-level only (enables `db.query` with `with: { relation: true }`). Foreign keys are DB-level constraints. **AFENDA requires both**: FKs for integrity, relations for query convenience.

### 4.8 Row-Level Security and Policies

- **Enable RLS**: `pgTable.withRLS("name", { ... })` (v1.0+).
- **Policies**: Adding any `pgPolicy()` to a table enables RLS automatically.
- **Define policies**: `pgPolicy("name", { as: 'permissive'|'restrictive', to: role, for: 'select'|'insert'|'update'|'delete'|'all', using: sql\`...\`, withCheck: sql\`...\` })` in the table's third argument.
- **Define roles**: `pgRole("name", { createRole, createDb, inherit })` or `.existing()` for pre-existing DB roles.
- **Link to existing tables**: `pgPolicy("name", { ... }).link(existingTable)` for tables you don't own.
- **drizzle.config.ts**: Set `entities: { roles: true }` to include role management in migrations; use `provider: 'neon'|'supabase'` to exclude provider-defined roles.
- **Neon helpers**: `crudPolicy({ role, read: true, modify: false })` from `drizzle-orm/neon` for shorthand CRUD policies.
- **RLS on views**: use `.with({ securityInvoker: true })` on `pgView` to enforce caller's RLS policies.

**AFENDA convention**: RLS is opt-in per ADR; when enabled, use the full pattern with `afenda.tenant_id` session variable (see Section 6.5).

**Current codebase (AFENDA-HYBRID):** **No row-level security policies are defined yet.** There are no `pgPolicy()`, `withRLS`, or `CREATE POLICY` / `ALTER TABLE … ENABLE ROW LEVEL SECURITY` statements in `packages/db/src/schema-platform` (or `schema-hrm`) or migrations. Tenant isolation today relies on **application queries**, **foreign keys**, and **triggers** (e.g. same-tenant parent checks). `pnpm check:rls-policies` emits **info**-level reminders only. To see policies in PostgreSQL or Drizzle Studio, add RLS via schema + migration (or custom SQL with a `CUSTOM:` block) after an ADR.

### 4.9 Views and Materialized Views

- **Views**: `pgView("name").as((qb) => qb.select(...).from(...))` — column schema auto-inferred.
- **Views with raw SQL**: `pgView("name", { col1: integer(), ... }).as(sql\`...\`)` — declare column schema.
- **View options**: `.with({ checkOption: 'cascaded', securityBarrier: true, securityInvoker: true })`
- **Existing views (read-only)**: `pgView("name", { ... }).existing()` — Drizzle Kit won't generate DDL.
- **Materialized views**: `pgMaterializedView("name").as((qb) => ...)` with `.using('btree')`, `.with({ fillfactor: 90, autovacuum_enabled: true })`, `.tablespace('name')`, `.withNoData()`
- **Refresh**: `await db.refreshMaterializedView(view)`, `.concurrently()`, `.withNoData()`

**Limitation**: Indexes on materialized views require custom SQL. After running `drizzle-kit generate`, append to the migration file:

```sql
-- CUSTOM: Index on materialized view for fast lookups
CREATE INDEX CONCURRENTLY idx_mv_sales_tenant_date
  ON finance.mv_sales_summary (tenant_id, sale_date);
```

### 4.10 Sequences (`pgSequence`)

- **Public schema**: `pgSequence("name", { startWith, increment, maxValue, minValue, cache, cycle })`
- **Named schema**: `coreSchema.sequence("name", { ... })`

Use when you need controlled ID generation outside of `generatedAlwaysAsIdentity` (e.g., invoice numbers, batch IDs).

### 4.11 PostgreSQL-Specific Data Types

Type selection guide for common AFENDA needs:

**Numeric precision (financial data)**

- `numeric(precision, scale)` via `numeric({ precision: 12, scale: 2 })` — **must** be used for all monetary columns in the `finance` schema. Never use `real`/`doublePrecision` for money. Maps to `z.string()` in Zod (numeric is returned as string for precision).

**Date vs timestamp**

- `date()` — calendar dates only (birthdays, hire dates, fiscal periods).
- `timestamp({ withTimezone: true })` — points in time (events, audit). **AFENDA default** for timestamps.
- `time({ withTimezone: true })` — for schedules only.
- `interval()` — durations (SLA, retention). Maps to `z.string()`.

**Network types** (security/audit)

- `inet()` — IPv4/IPv6 host addresses (login IP tracking).
- `cidr()` — network blocks (IP allowlists).
- `macaddr()` / `macaddr8()` — device tracking.

**Semi-structured (JSONB)**

- `jsonb()` — flexible metadata, tenant settings, feature flags. Prefer over `json` (indexing, containment).
- GIN index: `index("idx_settings").using("gin", t.settings)` for `@>` containment queries.
- **AFENDA**: Use JSONB for `tenantSettings`, `userPreferences`, `featureFlags`; type at the Zod layer:

```typescript
const tenantSelectSchema = createSelectSchema(tenants, {
  settings: z.object({ theme: z.string(), locale: z.string() }),
});
```

**Full-text search**

PostgreSQL's full-text search uses `tsvector` (indexed document) and `tsquery` (search query). Use Drizzle `customType` or raw SQL.

Pattern: Add a stored generated column + GIN index + optional trigger for complex cases.

Example (append to generated migration):

```sql
-- CUSTOM: Add tsvector column as stored generated column
ALTER TABLE hr.job_postings
  ADD COLUMN search_vector tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('english', coalesce(description, '')), 'B') ||
    setweight(to_tsvector('english', coalesce(requirements, '')), 'C')
  ) STORED;

-- CUSTOM: GIN index for fast full-text search
CREATE INDEX idx_job_postings_search
  ON hr.job_postings USING gin(search_vector);

-- Query example (in application code):
-- SELECT * FROM hr.job_postings
-- WHERE search_vector @@ to_tsquery('english', 'engineer & (senior | lead)');
```

For complex cases where generated columns can't capture the logic, use a trigger instead (see trigger examples in Section 4.13).

**Generated columns**

- **Stored**: `.generatedAlwaysAs(sql\`expression\`).stored()` — computed on write.
- **Virtual** (PG 18+): `.generatedAlwaysAs(sql\`expression\`)` — computed on read.
- Use cases: `fullName` from first+last, `searchVector` from title+description, `totalAmount` from subtotal+tax.

**UUID vs integer PK**

- `uuid()` with `.defaultRandom()` — uses `gen_random_uuid()` (PG 13+). Use when entities are referenced across systems or exposed in URLs.
- **AFENDA default**: `integer().generatedAlwaysAsIdentity()` for PKs; reserve `uuid()` for cross-system or URL-stable identifiers.

### 4.12 PostgreSQL Extensions

- **pg_vector**: `vector({ dimensions: 3 })`; HNSW/IVFFlat via `.using('hnsw', t.embedding.op('vector_cosine_ops'))`; distance helpers (`l2Distance`, `cosineDistance`, etc.).
- **PostGIS**: `geometry({ type: 'point', mode: 'xy' })`; GiST via `.using('gist', t.geo)`.
- **pgcrypto**: `sql\`gen_random_uuid()\``in defaults;`pgp_sym_encrypt`/`pgp_sym_decrypt` for PII encryption.
- **pg_trgm**: Trigram similarity; GIN/GiST with `gin_trgm_ops`.
- **Config**: `extensionsFilters: ["postgis"]` in `drizzle.config.ts` to exclude extension system tables from introspection.

### 4.13 Custom SQL Escapes

Features Drizzle does **not** natively support — add custom SQL to the generated migration file and comment it as `-- CUSTOM: <purpose>`.

**Triggers** (audit / timestamps)

- `BEFORE INSERT/UPDATE` for `updated_at = now()`.
- `AFTER INSERT/UPDATE/DELETE` for writing to `audit.audit_trail`.
- Pattern: Create the trigger function once in a base migration, then attach triggers per table.

Example (append to generated migration):

```sql
-- CUSTOM: Trigger function for updated_at (create once)
CREATE OR REPLACE FUNCTION core.set_updated_at()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = now();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- CUSTOM: Attach trigger to specific table
CREATE TRIGGER trg_employees_updated_at
  BEFORE UPDATE ON hr.employees
  FOR EACH ROW
  EXECUTE FUNCTION core.set_updated_at();
```

For audit trail triggers:

```sql
-- CUSTOM: Audit trail trigger function (create once)
CREATE OR REPLACE FUNCTION audit.log_change()
RETURNS TRIGGER AS $$
BEGIN
  INSERT INTO audit.audit_trail (
    tenant_id, table_name, operation, row_id,
    old_data, new_data, changed_by, changed_at
  ) VALUES (
    COALESCE(NEW.tenant_id, OLD.tenant_id),
    TG_TABLE_SCHEMA || '.' || TG_TABLE_NAME,
    TG_OP,
    COALESCE(NEW.employee_id, OLD.employee_id),
    CASE WHEN TG_OP IN ('UPDATE', 'DELETE') THEN row_to_json(OLD) END,
    CASE WHEN TG_OP IN ('INSERT', 'UPDATE') THEN row_to_json(NEW) END,
    current_setting('afenda.user_id', true)::int,
    now()
  );
  RETURN COALESCE(NEW, OLD);
END;
$$ LANGUAGE plpgsql;

-- CUSTOM: Attach audit trigger to specific table
CREATE TRIGGER trg_employees_audit
  AFTER INSERT OR UPDATE OR DELETE ON hr.employees
  FOR EACH ROW
  EXECUTE FUNCTION audit.log_change();
```

**Table partitioning**

Partitioning improves query performance and maintenance for large tables. PostgreSQL 18 supports range, list, and hash partitioning.

- **Range**: Partition by date/timestamp for time-series data (most common for AFENDA).
- **List**: Partition by discrete values (e.g., `tenant_id` for large-tenant isolation).
- **Hash**: Even distribution across partitions.

**AFENDA convention**: Partition `audit.audit_trail` and large `*.operations` tables by range on timestamp when row count is expected to exceed ~10M rows.

Example (append to generated migration):

```sql
-- CUSTOM: Create partitioned audit trail table
CREATE TABLE audit.audit_trail (
  audit_id bigint GENERATED ALWAYS AS IDENTITY,
  tenant_id integer NOT NULL,
  table_name text NOT NULL,
  operation text NOT NULL,
  row_id integer,
  old_data jsonb,
  new_data jsonb,
  changed_by integer,
  changed_at timestamptz NOT NULL DEFAULT now(),
  PRIMARY KEY (audit_id, changed_at)
) PARTITION BY RANGE (changed_at);

-- CUSTOM: Create initial partitions (quarterly)
CREATE TABLE audit.audit_trail_2026_q1
  PARTITION OF audit.audit_trail
  FOR VALUES FROM ('2026-01-01') TO ('2026-04-01');

CREATE TABLE audit.audit_trail_2026_q2
  PARTITION OF audit.audit_trail
  FOR VALUES FROM ('2026-04-01') TO ('2026-07-01');

-- CUSTOM: Index on each partition
CREATE INDEX idx_audit_trail_2026_q1_tenant
  ON audit.audit_trail_2026_q1 (tenant_id, changed_at);

-- CUSTOM: Automated partition creation function (optional)
CREATE OR REPLACE FUNCTION audit.create_next_quarter_partition()
RETURNS void AS $$
DECLARE
  next_quarter_start date;
  next_quarter_end date;
  partition_name text;
BEGIN
  next_quarter_start := date_trunc('quarter', now() + interval '3 months');
  next_quarter_end := next_quarter_start + interval '3 months';
  partition_name := 'audit_trail_' || to_char(next_quarter_start, 'YYYY_q') || extract(quarter from next_quarter_start)::text;

  EXECUTE format(
    'CREATE TABLE IF NOT EXISTS audit.%I PARTITION OF audit.audit_trail FOR VALUES FROM (%L) TO (%L)',
    partition_name, next_quarter_start, next_quarter_end
  );
END;
$$ LANGUAGE plpgsql;
```

Benefits: Partition pruning (faster queries), easier archival (detach old partitions), parallel maintenance.

**Stored functions/procedures**

- Use for complex DB-side logic, transaction-boundary audit, custom aggregates. **AFENDA**: Prefer application logic; use sparingly for audit triggers and MV refresh.

**Exclusion constraints**

Prevent overlapping ranges (schedules, bookings, validity ranges). Requires the `btree_gist` extension.

Example (append to generated migration):

```sql
-- CUSTOM: Enable btree_gist extension (run once)
CREATE EXTENSION IF NOT EXISTS btree_gist;

-- CUSTOM: Exclusion constraint for room bookings
ALTER TABLE hr.room_bookings
  ADD CONSTRAINT excl_no_room_overlap
  EXCLUDE USING gist (
    room_id WITH =,
    time_range WITH &&
  );

-- CUSTOM: Exclusion constraint for employee schedules (with tenant isolation)
ALTER TABLE hr.employee_schedules
  ADD CONSTRAINT excl_no_schedule_overlap
  EXCLUDE USING gist (
    tenant_id WITH =,
    employee_id WITH =,
    shift_range WITH &&
  );
```

**AFENDA convention**: Use for scheduling/booking domain tables where range overlap must be forbidden at the database level.

**Domains** (custom constrained types)

Domains define reusable types with built-in constraints. Use when the same validation must be enforced across many tables at the database level.

Example (append to generated migration):

```sql
-- CUSTOM: Domain for email addresses
CREATE DOMAIN core.email_address AS text
  CHECK (VALUE ~ '^[^@]+@[^@]+\.[^@]+$');

-- CUSTOM: Domain for phone numbers (E.164 format)
CREATE DOMAIN core.phone_number AS text
  CHECK (VALUE ~ '^\+[1-9]\d{1,14}$');

-- CUSTOM: Domain for positive amounts
CREATE DOMAIN finance.positive_amount AS numeric(12,2)
  CHECK (VALUE > 0);
```

Usage in Drizzle schema (after domain is created):

```typescript
import { customType } from "drizzle-orm/pg-core";

const emailAddress = customType<{ data: string }>({
  dataType() {
    return "core.email_address";
  },
});

export const users = securitySchema.table("users", {
  userId: integer().primaryKey().generatedAlwaysAsIdentity(),
  email: emailAddress().notNull().unique(),
});
```

**AFENDA convention**: Use domains sparingly — prefer Zod validation at the application layer. Reserve domains for cross-table constraints that must be enforced at the database level (e.g., email format, phone format, positive amounts).

**Pattern**: Run `drizzle-kit generate`, then append the custom SQL block to the generated `.sql` file before committing.

### 5.2 Zod Schema Generation (`drizzle-orm/zod`)

Use the **built-in** integration from `drizzle-orm/zod` (not the deprecated `drizzle-zod` package). Prescribe **Zod 4** (`zod@^4.0.0`).

- **Imports:** in the same module as `createSelectSchema` / `createInsertSchema` / `createUpdateSchema`, use `import { z } from "zod/v4"`. Drizzle’s typings reference `zod/v4`; `import { z } from "zod"` is a separate declaration path in Zod 4 and breaks assignability on refinements (TS2322 / “two different types with this name exist”).
- `createSelectSchema(table)` — validates data from the DB; works on tables, views, enums.
- `createInsertSchema(table)` — validates insert payloads; respects defaults/generated columns.
- `createUpdateSchema(table)` — validates partial updates (all fields optional).
- **Refinements**: second argument with column overrides:
  - Callback to extend: `{ name: (schema) => schema.max(100) }`
  - Zod schema to overwrite: `{ preferences: z.object({ theme: z.string() }) }`
- **Factory**: `createSchemaFactory({ zodInstance: extendedZ, coerce: { date: true } })` for extended Zod or coercion.
- **Pattern**: Export `insertSchema`, `selectSchema`, `updateSchema` alongside each table:

```typescript
import { createSelectSchema, createInsertSchema, createUpdateSchema } from "drizzle-orm/zod";
import { z } from "zod/v4";
import { tenants } from "./tenants";

export const tenantSelectSchema = createSelectSchema(tenants);
export const tenantInsertSchema = createInsertSchema(tenants, {
  tenantCode: (schema) => schema.max(50),
});
export const tenantUpdateSchema = createUpdateSchema(tenants);
```

## 6 Tenancy and Isolation

### 6.1 Default Model: tenant_id Column

The default tenancy strategy is a **shared-schema, shared-table** model using a `tenantId` discriminator column (mapped to `tenant_id` in DB via `casing: 'snake_case'`).

**Requirements:**

- Every tenant-scoped table **must** include `tenantId` as an explicit column with an explicit `foreignKey()` definition.
- Do **not** use the deprecated `tenantScopedColumns` mixin — define `tenantId` explicitly for precise FK control.
- Index `tenantId` as the leading column (Section 4.4): `index("idx_{table}_tenant").on(t.tenantId)`.
- All uniqueness constraints on tenant-scoped data **must** include `tenantId` (e.g., `unique().on(t.tenantId, t.orgCode)`).
- Application queries **must** always filter by `tenantId`; never rely on the absence of a filter.

**Standard pattern:**

```typescript
import { integer, foreignKey } from "drizzle-orm/pg-core";
import { tenants } from "../../core/tenants";

export const myTable = mySchema.table(
  "my_table",
  {
    myTableId: integer().primaryKey().generatedAlwaysAsIdentity(),
    tenantId: integer().notNull(),
    // ... other columns
    ...timestampColumns,
    ...softDeleteColumns,
    ...auditColumns,
  },
  (t) => [
    index("idx_my_table_tenant").on(t.tenantId),
    foreignKey({
      columns: [t.tenantId],
      foreignColumns: [tenants.tenantId],
      name: "fk_my_table_tenant",
    })
      .onDelete("restrict")
      .onUpdate("cascade"),
  ]
);
```

### 6.2 When to Use Row-Level Security (RLS)

RLS is **not** the default. Enable it only when:

- A tenant has a contractual or regulatory requirement for database-enforced isolation.
- A high-security classification mandates defence-in-depth beyond application-layer filtering.

Before enabling RLS, the team **must** produce an ADR documenting:

- The specific regulatory or contractual driver.
- Operational cost (query plan impact, debugging complexity, connection-pool implications).
- Rollback strategy if RLS causes performance regressions.

### 6.3 Tenancy Options Comparison

| Option             | Strength                        | Trade-off                                        |
| ------------------ | ------------------------------- | ------------------------------------------------ |
| `tenant_id` column | Simple; easy to index and query | Requires application discipline to always filter |
| Row-Level Security | Database-enforced isolation     | Operational complexity; harder to debug          |
| Schema-per-tenant  | Strongest isolation boundary    | Does not scale beyond a small number of tenants  |

### 6.4 Session Context Pattern

Applications **must** set PostgreSQL session variables at the start of every database session (or transaction, depending on pooling mode). Use the shared helper **`packages/db/src/_session/setSessionContext.ts`** (re-exported from `schema-platform/index.ts`) instead of duplicating raw SQL. Conceptual shape:

```typescript
import { sql } from "drizzle-orm";

export async function setSessionContext(
  db: { execute: (query: unknown) => Promise<unknown> },
  ctx: { tenantId: number; userId?: number }
) {
  await db.execute(sql`SET LOCAL afenda.tenant_id = ${ctx.tenantId.toString()}`);
  if (ctx.userId != null) {
    await db.execute(sql`SET LOCAL afenda.user_id = ${ctx.userId.toString()}`);
  }
}
```

These variables are consumed by:

- **Audit triggers** — to record who made the change.
- **RLS policies** — when RLS is enabled (see Section 4.8 for `pgPolicy` + `pgRole`).
- **Observability** — to correlate database activity back to API requests.

### 6.5 Drizzle RLS Pattern (opt-in)

When RLS is adopted per ADR, use `pgPolicy` + `pgRole` with the `afenda.tenant_id` session variable:

```typescript
import { pgPolicy, pgRole, sql } from "drizzle-orm/pg-core";

const appRole = pgRole("app_write").existing();

export const orders = hrSchema
  .table(
    "orders",
    {
      orderId: integer().primaryKey().generatedAlwaysAsIdentity(),
      tenantId: integer().notNull(),
      // ...other columns
    },
    (t) => [
      pgPolicy("tenant_isolation_select", {
        as: "permissive",
        to: appRole,
        for: "select",
        using: sql`${t.tenantId} = current_setting('afenda.tenant_id')::int`,
      }),
      pgPolicy("tenant_isolation_insert", {
        as: "permissive",
        to: appRole,
        for: "insert",
        withCheck: sql`${t.tenantId} = current_setting('afenda.tenant_id')::int`,
      }),
    ]
  )
  .withRLS();
```

## 7 Integrity and Constraint Patterns

### 7.1 Constraint Hierarchy

Apply constraints in this order of preference. See **Section 4.5** for Drizzle syntax.

1. **NOT NULL** — every column is `.notNull()` unless there is a documented reason for nullability.
2. **CHECK** — for domain-valid ranges and enumerations via `check("chk_name", sql\`...\`)`.
3. **UNIQUE** — for natural-key uniqueness; always include `tenantId` on tenant-scoped tables (`.unique()` or `unique().on(t.tenantId, t.orgCode)`).
4. **Foreign keys** — for referential integrity (Section 4.3).
5. **Triggers** — only when declarative constraints cannot express the invariant (Section 4.13).

### 7.2 Soft Deletes vs. Hard Deletes

- Prefer a `deletedAt` (mapped to `deleted_at`) soft-delete pattern for auditability.
- Use a partial unique index so uniqueness applies only to active records (Drizzle):

```typescript
uniqueIndex("uq_active_org_code")
  .on(t.tenantId, t.orgCode)
  .where(sql`${t.deletedAt} IS NULL`);
```

- Hard deletes are acceptable for transient operational data (e.g., job queues) where audit requirements are met by the `audit` schema.

## 8 Migrations: Versioning and Patterns

### 8.1 Strategy: Drizzle Kit (Generate + Migrate)

Use **Option 3** from Drizzle docs: `drizzle-kit generate` produces SQL migration files; commit them to Git. Apply via `drizzle-kit migrate` or runtime `migrate()` from `drizzle-orm/node-postgres/migrator` (or your driver's migrator).

**Commands**

| Command                | Purpose                                                                          |
| ---------------------- | -------------------------------------------------------------------------------- |
| `drizzle-kit generate` | Generate migration SQL from schema changes (run after editing `schema/**/*.ts`). |
| `drizzle-kit migrate`  | Apply pending migrations to the database.                                        |
| `drizzle-kit push`     | Dev only — push schema directly; do not use for production.                      |
| `drizzle-kit pull`     | Introspect existing DB into schema (use for legacy or one-off sync).             |
| `drizzle-kit check`    | Verify migration consistency.                                                    |

**File location**: Migrations live under the path configured as `out` in `drizzle.config.ts` (e.g. `./packages/db/src/migrations`). Use `migrations.prefix: "timestamp"` for timestamp-based filenames.

### 8.2 Writing Migrations

**Workflow**

1. Change table/enum definitions in `packages/db/src/schema-platform/**/*.ts`.
2. Run `drizzle-kit generate`; review the generated `.sql` file.
3. If needed, append **custom SQL** (triggers, partitioning, exclusion constraints, etc.) with a clear `-- CUSTOM: <purpose> (CSQL-XXX)` marker (Section 4.13 and 8.2.1).
4. Commit the new migration file.

**Atomicity**

- Generated migrations run in a transaction by default.
- Long-running DDL (e.g. `CREATE INDEX CONCURRENTLY`) cannot run inside a transaction: use a separate migration and document in the header.

**Two-Phase Change Pattern**

Non-trivial column or constraint changes follow a two-phase approach:

| Phase | Step                                                                               | What happens                             |
| ----- | ---------------------------------------------------------------------------------- | ---------------------------------------- |
| 1     | Add column as nullable in schema; generate migration; deploy app that dual-writes. | Add column as nullable.                  |
| 2     | Change column to `.notNull()`, add defaults/constraints; generate migration.       | Backfill, set NOT NULL, add constraints. |

#### 8.2.1 Custom SQL Rules

**Principle**: Drizzle ORM is the single source of truth for schema management. Custom SQL is permitted **only** for advanced PostgreSQL features that cannot be expressed in Drizzle's schema language.

**Allowed Custom SQL (requires approval and documentation):**

| Feature                   | Example                                            | Why Drizzle Can't Express                            |
| ------------------------- | -------------------------------------------------- | ---------------------------------------------------- |
| **Table Partitioning**    | `PARTITION BY RANGE (created_at)`                  | Drizzle doesn't support partition syntax             |
| **Partition Creation**    | `CREATE TABLE ... PARTITION OF`                    | Partition definitions require explicit SQL           |
| **Exclusion Constraints** | `EXCLUDE USING gist (...)`                         | Not supported in Drizzle constraint API              |
| **Triggers**              | `CREATE TRIGGER trg_audit ...`                     | Trigger definitions require PL/pgSQL                 |
| **Trigger Functions**     | `CREATE FUNCTION log_change() RETURNS trigger`     | Function bodies require PL/pgSQL                     |
| **Stored Procedures**     | `CREATE PROCEDURE process_payroll(...)`            | Procedure bodies not supported                       |
| **Advanced Indexes**      | `CREATE INDEX ... USING gin (data jsonb_path_ops)` | Operator classes like `jsonb_path_ops` not supported |
| **RLS Policies**          | `CREATE POLICY tenant_isolation ...`               | Policy definitions require SQL                       |
| **Partition Maintenance** | `CREATE FUNCTION create_next_partition()`          | Maintenance functions require PL/pgSQL               |

**Forbidden Custom SQL (must use Drizzle):**

| Feature                      | Use Drizzle Instead                     |
| ---------------------------- | --------------------------------------- |
| Table definitions            | `pgTable()` with column definitions     |
| Column types and constraints | `.notNull()`, `.default()`, `.unique()` |
| Foreign keys                 | `.references()` or `foreignKey()`       |
| Basic indexes                | `.index()` on table definition          |
| Check constraints            | `check()` constraint helper             |
| Unique constraints           | `.unique()` or `uniqueIndex()`          |
| Enums                        | `pgEnum()`                              |

**Custom SQL Format Requirements:**

1. **Marker**: Every custom SQL block must start with:

   ```sql
   -- CUSTOM: <purpose> (CSQL-XXX)
   ```

   - `<purpose>`: Human-readable description (e.g., "Add audit trigger for employees table")
   - `CSQL-XXX`: Unique reference ID (e.g., CSQL-010)

2. **Placement**: Custom SQL must be at the **end** of the migration file, after all Drizzle-generated SQL

3. **Registry Entry**: Add entry to `packages/db/src/schema-platform/audit/CUSTOM_SQL_REGISTRY.json`:

   ```json
   "CSQL-XXX": {
     "purpose": "Description",
     "migration": "20260320120000_migration_name",
     "type": "TRIGGER|FUNCTION|PARTITION|etc",
     "justification": "Why Drizzle can't express this",
     "rollback": "How to rollback",
     "approvedBy": "dba-team",
     "approvedDate": "2026-03-20",
     "sqlLines": "45-67"
   }
   ```

4. **Approval**: Create GitHub issue using `.github/ISSUE_TEMPLATE/custom-sql-request.md` template and get DBA approval before implementation

**Compliant Example:**

```sql
-- (Drizzle-generated SQL above)
--> statement-breakpoint
ALTER TABLE "hr"."employees" ADD CONSTRAINT "fk_employees_tenant"
  FOREIGN KEY ("tenantId") REFERENCES "core"."tenants"("tenantId");
--> statement-breakpoint

-- CUSTOM: Add audit trigger for employees table (CSQL-010)
CREATE TRIGGER trg_employees_audit
  AFTER INSERT OR UPDATE OR DELETE ON hr.employees
  FOR EACH ROW
  EXECUTE FUNCTION audit.log_change_7w1h();
```

**Non-Compliant Examples:**

```sql
-- ❌ Wrong: Hand-written migration (no Drizzle-generated SQL)
CREATE TABLE core.new_table (
  id serial PRIMARY KEY,
  name text NOT NULL
);

-- ❌ Wrong: Custom SQL without marker
CREATE TRIGGER trg_audit ...

-- ❌ Wrong: Custom SQL in middle of file (should be at end)
CREATE TABLE ... (Drizzle-generated)
CREATE TRIGGER ... (custom, unmarked)
ALTER TABLE ... (more Drizzle-generated)

-- ❌ Wrong: Using custom SQL for something Drizzle can express
-- CUSTOM: Add foreign key (CSQL-011)
ALTER TABLE hr.employees ADD CONSTRAINT fk_employees_dept
  FOREIGN KEY (department_id) REFERENCES hr.departments(department_id);
-- Should use: .references(() => departments.departmentId) in schema
```

#### 8.2.2 Migration Validation

All migrations are automatically validated by CI to ensure compliance with schema lockdown rules.

**Validation Checks:**

| Check                        | Script (under `packages/db/scripts/`) | Validates                                                   |
| ---------------------------- | ------------------------------------- | ----------------------------------------------------------- |
| **Format Validation**        | `scripts/ci/validate-migrations.ts`   | Snapshot.json + migration.sql pairs exist                   |
| **Checksum Validation**      | `scripts/ci/validate-migrations.ts`   | Detects manual edits to Drizzle-generated SQL               |
| **Snapshot Alignment**       | `scripts/ci/validate-migrations.ts`   | SQL operations match snapshot state transitions             |
| **Custom SQL Markers**       | `scripts/ci/validate-migrations.ts`   | All custom SQL has `-- CUSTOM: <purpose> (CSQL-XXX)` format |
| **Registry Cross-Reference** | `scripts/ci/validate-migrations.ts`   | CSQL-XXX IDs exist in CUSTOM_SQL_REGISTRY.json              |
| **Custom SQL Placement**     | `scripts/ci/validate-migrations.ts`   | Custom SQL is at end of file                                |
| **Schema Drift**             | `scripts/ci/detect-schema-drift.ts`   | No uncommitted schema changes                               |

**Running Validation Locally:**

```bash
# Quick migration validation only (`scripts/ci/validate-migrations.ts` honors `--quick`)
pnpm check:migrations --quick

# Full validation
pnpm check:migrations
pnpm check:drift

# Run all gates (recommended before committing)
pnpm gate:early
```

**Bypass Flags (use with caution):**

```bash
# Allow drift during prototyping (warns but doesn't fail)
pnpm check:drift --allow-drift

# Bypass migration validation (logs to audit, emergency only)
pnpm check:migrations --bypass

# Skip drift check in git commit
SKIP_DRIFT_CHECK=1 git commit -m "..."

# Skip migration check in git commit
SKIP_MIGRATION_CHECK=1 git commit -m "..."
```

**CI Enforcement:**

- **Pre-commit hook** (`.husky/pre-commit`): Runs quick validation before commit
- **Early Gate workflow** (`.github/workflows/early-gate.yml`): Runs validation on PR (see **§9.3** for scope and blocking vs non-blocking jobs)
- **DB CI workflow** (`.github/workflows/db-ci.yml`): Path-filtered DB pipeline (migrations, custom SQL, tests); see **§9.3**
- **Daily scheduled job**: Detects drift even when no PRs are open

**Validation Failures:**

If validation fails, check the error message:

| Error                                       | Cause                                   | Fix                                    |
| ------------------------------------------- | --------------------------------------- | -------------------------------------- |
| `Missing snapshot.json`                     | Migration folder incomplete             | Regenerate with `drizzle-kit generate` |
| `Invalid custom SQL marker format`          | Marker doesn't match pattern            | Use `-- CUSTOM: <purpose> (CSQL-XXX)`  |
| `Custom SQL CSQL-XXX not found in registry` | Missing registry entry                  | Add entry to CUSTOM_SQL_REGISTRY.json  |
| `Registry migration mismatch`               | Registry references wrong migration     | Update registry entry                  |
| `Custom SQL should be at end of file`       | Custom SQL in wrong location            | Move to end after Drizzle SQL          |
| `Schema drift detected`                     | Schema files modified without migration | Run `pnpm db:generate`                 |
| `Migration SQL was manually edited`         | Hand-edited migration file              | Regenerate migration                   |

See [docs/SCHEMA_LOCKDOWN.md](../SCHEMA_LOCKDOWN.md) for complete validation reference.

### 8.3 Migration Process

1. **Feature branch** — Author updates schema TS files, runs `drizzle-kit generate`, adds custom SQL if needed, and documents rollback.
2. **CI (PR)** — Spin ephemeral Postgres, run `drizzle-kit migrate` (or runtime migrator) up then down then up; run unit and contract tests.
3. **Staging** — Merge triggers migration on staging; run smoke and performance checks.
4. **Production** — Deploy in maintenance window with monitoring; maintain rollback window.

### 8.4 Rollback Rules

- **Prefer forward-compatible changes.** Reverse via a subsequent forward migration when possible.
- **Destructive changes** (dropping columns, tables, constraints) require a documented rollback migration in the same PR and a restore-drill plan exercised in staging (at least quarterly).

### 9.2 CI Pipeline Requirements

The CI pipeline **must** perform the following steps on every pull request that touches `packages/db/src/schema-platform/` or `packages/db/src/migrations/`:

1. Spin up an ephemeral PostgreSQL instance (matching the production major version).
2. Run all migrations from zero to head (`drizzle-kit migrate` or runtime migrator).
3. Run all migrations back to zero (rollback if supported, or use a fresh DB for down test).
4. Re-apply all migrations to head.
5. Execute unit tests and contract tests against the resulting schema.
6. Run smoke queries.
7. Publish the migration artifact (versioned tarball or container layer).

Staging promotion is gated by passing smoke tests and performance benchmarks.
Production deployment requires a signed-off runbook and active monitoring hooks.

### 9.5 Smoke Test Template

Create `packages/db/src/__tests__/smoke.test.ts`:

```typescript
import { describe, it, expect, beforeAll, afterAll } from "vitest";
import { db } from "../db";
import { tenants } from "../schema-platform/core/tenants";
import { sql } from "drizzle-orm";

describe("Database Smoke Tests", () => {
  beforeAll(async () => {
    // Verify connection
    await db.execute(sql`SELECT 1`);
  });

  it("smoke: can query core.tenants", async () => {
    const result = await db.select().from(tenants).limit(1);
    expect(Array.isArray(result)).toBe(true);
  });

  it("smoke: can insert and select tenant", async () => {
    const [inserted] = await db
      .insert(tenants)
      .values({ tenantCode: `TEST_${Date.now()}`, name: "Test Tenant" })
      .returning();

    expect(inserted.tenantId).toBeDefined();

    const [selected] = await db
      .select()
      .from(tenants)
      .where(sql`${tenants.tenantId} = ${inserted.tenantId}`);

    expect(selected.tenantCode).toBe(inserted.tenantCode);

    // Cleanup
    await db.delete(tenants).where(sql`${tenants.tenantId} = ${inserted.tenantId}`);
  });

  it("smoke: session context can be set", async () => {
    await db.execute(sql`SET LOCAL afenda.tenant_id = '1'`);
    const result = await db.execute(sql`SELECT current_setting('afenda.tenant_id', true)`);
    expect(result.rows[0].current_setting).toBe("1");
  });

  it("smoke: required extensions are installed", async () => {
    const extensions = await db.execute(sql`
      SELECT extname FROM pg_extension 
      WHERE extname IN ('btree_gist', 'pgcrypto')
    `);
    const extNames = extensions.rows.map((r: { extname: string }) => r.extname);
    expect(extNames).toContain("btree_gist");
    expect(extNames).toContain("pgcrypto");
  });
});
```

## 10 Operations

### 10.1 Production Migration Checklist

**Before migration:**

- [ ] Verify backups completed successfully within the last 24 hours.
- [ ] Check replica lag is below the defined threshold.
- [ ] Confirm no long-running transactions are active.
- [ ] Review index bloat metrics; reindex if necessary.
- [ ] Notify on-call team and stakeholders.

**During migration:**

- [ ] Monitor lock contention in `pg_stat_activity`.
- [ ] Watch replication lag on all replicas.
- [ ] Track long-running queries.
- [ ] Monitor audit table growth rate.

**After migration:**

- [ ] Run smoke queries against production.
- [ ] Verify replication health across all replicas.
- [ ] Confirm all alerting thresholds return to normal.
- [ ] Close the maintenance window and notify stakeholders.

### 10.2 Rollback Decision Framework

| Condition                                          | Action                                                      |
| -------------------------------------------------- | ----------------------------------------------------------- |
| Smoke queries fail after migration                 | Execute rollback migration immediately.                     |
| Replication lag exceeds threshold during migration | Pause; assess root cause before proceeding or rolling back. |
| No issues after rollback window closes             | Mark migration as stable; archive rollback script.          |

## 11 Security

### 11.1 Credential Management

- Store all database credentials in a secrets vault (e.g., HashiCorp Vault, AWS Secrets Manager).
- Rotate credentials on a defined schedule (minimum quarterly).
- Never store credentials in source code, environment files committed to Git, or migration scripts.

### 11.2 Role-Based Access

Define and enforce least-privilege database roles:

| Role        | Permissions                                                  | Used by                                                 |
| ----------- | ------------------------------------------------------------ | ------------------------------------------------------- |
| `app_write` | `SELECT`, `INSERT`, `UPDATE`, `DELETE` on application tables | Application services (write path)                       |
| `app_read`  | `SELECT` on application tables                               | Application services (read path), read replicas         |
| `reporting` | `SELECT` on reporting views and materialized views           | BI tools, dashboards                                    |
| `migration` | Full DDL privileges                                          | CI/CD pipeline only; never used by running applications |

### 11.3 Personally Identifiable Information (PII)

- Maintain a PII registry that documents every column containing PII.
- Columns requiring encryption at rest must use PostgreSQL's `pgcrypto` extension or application-layer encryption with key management.
- Columns requiring tokenization must document the tokenization provider and lookup mechanism.
- Access to PII columns is restricted to roles with explicit PII-access grants.

### Drizzle primitives at a glance

- **Schema**: `pgSchema("name")` → `schema.table("t", cols, tableConfig)`
- **PK**: `.primaryKey().generatedAlwaysAsIdentity()` or `primaryKey({ columns: [...] })`
- **FK**: `.references(() => ref.column)` or `foreignKey({ columns, foreignColumns })`
- **Index**: `index("name").on(t.col)`, `uniqueIndex("name").on(t.col)`, `.where()`, `.using("gin", ...)`, `.concurrently()`
- **Constraints**: `.notNull()`, `.unique()`, `check("name", sql\`...\`)`, `.default()`, `.generatedAlwaysAs()`
- **RLS**: `pgTable.withRLS()`, `pgPolicy()`, `pgRole()`

### Zod 4 import

For schema modules that use **`drizzle-orm/zod`**, import `z` from **`"zod/v4"`** so refinements match Drizzle’s types. (The package root `"zod"` is fine for app-only code that does not mix with `createInsertSchema` / similar.) Use `createSelectSchema`, `createInsertSchema`, `createUpdateSchema` from `drizzle-orm/zod`.
